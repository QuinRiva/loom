# MDX visual-plan skill: ours vs. BuilderIO

**Objective.** Compare T3 Code's first-party `skills/mdx-visual-plan/SKILL.md`
against BuilderIO's upstream `visual-plan` skill (`SKILL.md`, `README.md`, and
the six `references/*.md`), and decide which differences are gaps worth closing
versus deliberate, correct-by-design divergences.

This is a **reimplementation, not a fork.** We kept BuilderIO's authoring
discipline and block vocabulary but replaced its entire delivery model: their
plan is a hosted artefact published through an MCP connector to
`plan.agent-native.com` and annotated in their web app; ours is a plain `.mdx`
file in the workspace, rendered by our own runtime renderer, annotated natively
(highlight a span → comment card → injected user turn). So "BuilderIO has X, we
don't" is only a _gap_ when X is authoring or rendering guidance; when X is part
of the hosted app/connector layer, dropping it is the whole point.

This document is itself authored in our MDX block vocabulary and compiles
through our real `compilePlanMdx` — it doubles as a dogfood of the renderer.

<Callout tone="success">
  **Verdict: strong parity on the parts that matter, one real content gap.** We carried over
  essentially all of BuilderIO's authoring discipline and the full document-block vocabulary (with
  our own native-annotation adaptations), and we dropped the hosted app/connector layer cleanly. The
  one genuine gap worth closing is **wireframe fidelity guidance** — BuilderIO's `wireframe.md`
  teaches a much richer HTML-mockup craft (renderer icons, skeleton states, full-width chrome /
  pinned bottom bars, before/after comparability) that our renderer mostly *can* honour but our
  SKILL never teaches. A few upstream features (rough.js sketch overlay, `data-icon` SVGs,
  `skeleton`) our renderer does not implement at all.
</Callout>

<Callout tone="info">
  **Block coverage is complete.** Every non-legacy BuilderIO block has a home in
  our vocabulary. Renames (`custom-html`→`HtmlBlock`) and native-slot
  containers (`<Column>`/`<Tab>`) are deliberate; `<Mermaid>` and the
  first-party `<Prototype>` are net additions. The only "missing" upstream blocks
  are legacy round-trip shims that would be dead weight here.
</Callout>

<Callout tone="warning">
  **The one block-schema divergence an author will feel: `<Diagram>`.** BuilderIO's
  `diagram` takes semantic `data.html` + `data.css` with renderer primitives
  (`.diagram-panel`, `.diagram-card`, `.diagram-node`, `[data-rough]`), so it can
  express layered/grouped/matrix/swimlane architecture pictures. Ours is a
  constrained `{ nodes, edges }` model with `x`/`y` as 0–100 percentages. For a
  simple spatial layout ours is fine, and we steer authors to `<Mermaid>` for
  auto-layout; but for a rich layered diagram our `<Diagram>` has a real
  expressiveness ceiling. This is **correct-by-design** (arbitrary HTML/CSS is a
  larger attack + layout surface than our locked model), but authors with
  BuilderIO habits will notice it.
</Callout>

## Axis 3 — Visual-surface guidance (the real gap)

BuilderIO devotes two long references — `wireframe.md` (303 lines) and
`canvas.md` (129 lines) — to the _craft_ of HTML mockups. Our renderer
implements the mechanical contract faithfully (the `SURFACE_PRESETS` in
`screen.tsx` are copied from BuilderIO's, and the full `--wf-*` token set and
`.wf-*`/`.primary`/`.accent` helper classes exist in `index.css`), but our SKILL
compresses all of `wireframe.md` into one short "Wireframe authoring contract"
section. Two things are true at once: our renderer **omits** some upstream
features entirely, and our skill **under-teaches** the ones it does support.

<Callout tone="risk">
  **Gap worth closing: fold a compact wireframe/canvas quality bar into the
  SKILL.** Authors following our skill today will produce mechanically valid but
  low-craft mockups: centred chrome bars instead of full-width, empty lower
  bands, before/after pairs that aren't comparable, `Before`/`After` pills baked
  into the HTML (BuilderIO explicitly forbids this — the column header is the
  only place the state name belongs). None of this needs renderer work; it is
  pure authoring guidance we chose not to port. This is the single highest-value
  gap.

The most _actionable_ sub-gap is **canvas spacing**: our SKILL shows
`<Artboard x y>` in board units with no spacing guidance at all, so an author
following it will place overlapping frames. BuilderIO gives hard numbers —
~2 board units per pixel, x-gap ≥ 1100 between `browser` frames, ≥ 1300 between
`desktop` frames, row y-gap ≥ 1400. Port those numbers verbatim.

</Callout>

<Callout tone="info">
  **Correct-by-design within axis 3.** The rough.js sketch overlay, `data-icon` SVG vocabulary, and
  `skeleton` state are genuine renderer omissions, but our SKILL does **not** claim them, so there
  is no correctness risk — only a fidelity ceiling. Whether to build them is a product decision (see
  Open Questions), not a skill-accuracy bug.
</Callout>

## Axis 4 — The deliberately-dropped hosted layer

We dropped BuilderIO's entire hosted stack: the Plan MCP connector
(`create-visual-plan`, `get-plan-feedback`, `update-visual-plan`,
`get-plan-blocks`, `export-visual-plan`, …), `plan.agent-native.com`, the
never-inline publish rule, connector/auth reconnect flows, hosted comment-anchor
interpretation (`resolutionTarget`, `consumedCommentIds`, detached-thread
reconciliation), local-files bridge mode, and visibility/sharing. All of it is
replaced by: write an `.mdx` file, tell the user the path, they annotate spans
natively, comments arrive as injected user turns.

<Callout tone="success">
  **Dropped cleanly — no dangling references.** Our SKILL never mentions an MCP tool, a hosted URL,
  `get-plan-blocks`, publish/export, connection recovery, or comment-anchor routing. It states
  positively that "there is no hosted app, no MCP connector, and no external account". The four
  upstream references that are wholly hosted-layer (`connection.md`, `local-files.md`, and the
  hosted halves of `canvas.md`) simply have no analogue, which is correct.
</Callout>

## Axis 5 — Accuracy to the implementation

Because our skill describes _our_ renderer, it is in several places **more**
accurate to its target than BuilderIO's is to theirs — most notably the MDX
attribute-encoding contract (scalar/number/boolean/JSON-literal, no free
`{expressions}`), the closed-registry security model, and the `id` →
`data-plan-block-id` anchoring behaviour, all of which are exact to our code.

One small imprecision to correct:

<Callout tone="warning">
  **Minor accuracy fix: the sanitiser is a Tailwind theme-class blocklist, not a general
  allowlist.** Our SKILL says of wireframe HTML: "do not rely on any other framework classes (they
  are stripped by the sanitiser)". In fact `sanitizeWireframeHtml` only strips classes matching a
  Tailwind colour/shadow-utility regex (`bg-*`, `text-*`, `shadow-*`, arbitrary `bg-[…]`, etc.);
  arbitrary non-Tailwind layout classes survive. The guidance ("don't rely on other classes") is
  safe advice, but the stated mechanism overstates what is stripped. Tighten the wording so an
  author isn't surprised that a stray class survived.
</Callout>

Otherwise no place in our SKILL claims a capability the renderer lacks: the
sketch/skeleton/icon features are absent from both the renderer and the skill,
so they cannot mislead.

## Gaps worth closing vs. correct-by-design

<Callout tone="decision">
  **Gaps worth closing (in priority order):**

1. **Wireframe/canvas quality bar** — port a compressed version of BuilderIO's
   `wireframe.md` craft into our SKILL: full-width/pinned chrome bars, fill the
   frame, before/after comparability with column-header labels (never baked-in
   pills), "modify don't redesign", zoom sub-surfaces, and — most actionable —
   **canvas board-unit spacing numbers** (x-gap ≥ 1100 browser / ≥ 1300 desktop,
   row y-gap ≥ 1400, ~2 units/px) so authors stop overlapping artboards.
   Highest value; no renderer work required.
2. **Sanitiser wording fix (axis 5)** — one-line correction so the theme-class
   stripping isn't described as a general allowlist.
3. **"Concrete first read"** — optionally add BuilderIO's "lead an abstract plan
   with one concrete example" rule (we lean on the visual surface for this today).
4. **`<Diagram>` expressiveness note** — one line telling authors our diagram is
   nodes/edges only and to reach for `<Mermaid>` (or prose) for layered/matrix
   pictures, so they don't expect BuilderIO's HTML/CSS diagram richness.
5. **Renderer features (product calls, not skill gaps):** `skeleton` state,
   `data-icon` SVG vocabulary, rough.js sketch overlay — build only if mockup
   fidelity becomes a priority.
   </Callout>

<Callout tone="info">
  **Correct-by-design differences (leave as-is):** the entire hosted
  app/connector/auth/sharing/local-files layer; hosted comment-anchor routing
  (replaced by native span→turn injection); `rich-text` as a JSON block (we use
  native MDX prose); legacy round-trip blocks (`kit-tree`, `code-tabs`,
  `implementation-map`, `legacy-wireframe`); the `custom-html`→`HtmlBlock`
  rename and native `<Column>`/`<Tab>` slot tags; and our net additions
  (`<Mermaid>`, first-party `<Prototype>`). The in-skill self-review pass is
  covered by our reviewer-sub-thread workflow rather than being omitted.
</Callout>
