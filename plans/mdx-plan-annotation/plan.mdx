# First-party MDX plan annotation

**Objective.** Let a coding agent hand back a plan as a rendered, annotatable
`.mdx` document instead of chat-only prose, so a reviewer can select any span,
leave an anchored comment, and have that comment injected back to the agent as a
normal user turn. **Done** means: opening a `plans/<slug>/plan.mdx` file shows
the rendered plan in the doc panel, selecting prose surfaces a **Comment**
affordance, the comment becomes a composer chip, and on send it serialises into
the user turn carrying the quoted passage and its section.

This plan reuses the existing composer review-comment plumbing
(`addReviewComment` → `appendReviewCommentsToPrompt`) rather than adding a new
comment store; the genuinely new delta is the **MDX renderer** and the
**rendered-DOM anchoring engine**. A few decisions remain open — see the form at
the end.

## Approach

The renderer compiles MDX at runtime in the browser with `@mdx-js/mdx`
`evaluate()` against a fixed component registry. A remark guard rejects
`import`/`export` and raw `{expression}` bodies, and MDX's own unknown-component
trap (caught by an error boundary) closes the surface — a stronger bound than the
current tag allow-list. Each top-level block is rendered with a stable
`data-plan-block-id`, and the whole document sits under one
`[data-plan-root]` container that the annotation layer attaches to.

### The anchor model

A comment stores a portable anchor, not a DOM offset, so it survives re-render.
Prose selections become a text-quote anchor with surrounding context for
disambiguation; a selection inside a non-prose block (code, diagram, table)
falls back to a whole-block `visual` anchor keyed on the block id.

The resolver re-finds the quoted text on every layout change and re-binds an
exact character `Range`; when the quote is gone the comment degrades to a clean
"detached" card rather than being dropped.

### Injection path

On send, each mdx-anchor comment is formatted into the injected user turn
alongside the section title and quoted passage — reusing
`appendReviewCommentsToPrompt`, the same seam that already carries line-based
code-review comments.

## Scope and non-goals

In scope: the 8 document blocks, prose annotation, whole-block annotation, and
composer-store injection. **Non-goals for this cut:** a `comments.json` sidecar
for external-tool interop (deferred), and the rich visual surfaces (wireframe
canvas, live prototype tabs, design-fidelity screens) — those are a later phase.
A future sync endpoint, if the sidecar is ever wanted, would look like this — it
is **not** built now:

<Endpoint

## Container blocks

Container blocks nest other plan blocks so a reviewer can compare or group
surfaces. `<Columns>` lays child blocks side by side (the common case is a
before/after pair); `<TabsBlock>` stacks child blocks behind a keyboard-accessible
tab strip. Both are MDX-native — each slot's blocks are ordinary children — and
each nested block stays independently annotatable.

## Live prototype

The **prototype** is a functional, operable mock that runs in a sandboxed
iframe (`allow-scripts allow-forms`, opaque origin, no `allow-same-origin`) under
a strict `srcdoc` CSP — it stays interactive but cannot reach the parent window,
cookies, or app RPCs. It carries `data-plan-block-type="prototype"`, so the
whole prototype is annotatable as one block (in-frame pins are a deferred tier).

## Embedded HTML

An `<HtmlBlock>` shows arbitrary **static** HTML in a fully locked iframe
(`sandbox=""` — no scripts run at all), so no sanitiser is needed on top of the
sandbox. It carries `data-plan-block-type="html"` and is annotatable as one block.
