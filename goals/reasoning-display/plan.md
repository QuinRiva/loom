# Plan: Reasoning (thinking) trace display — Option A, tri-state toggle

## Summary

Reasoning traces are already produced by every driver as
`content.delta { streamKind: "reasoning_text" }` (PiDriver maps pi's
`thinking_delta`; Codex/Claude adapters do the same). They die at
`ProviderRuntimeIngestion`, which only consumes `assistant_text`. There is **no**
existing implementation to port — t3code upstream drops reasoning identically
(its only reasoning reference is a truncated "Reasoning update" work-log row from
`task.progress`).

This plan adds a **parallel reasoning channel** that mirrors the assistant-text
pipeline end to end, and renders reasoning as an inline collapsible block above
the answer (Option A), gated by a tri-state client setting.

## Architecture decision: attach reasoning to the assistant message (Model 1)

Reasoning text is stored as two new fields on `OrchestrationMessage`
(`reasoningText`, `reasoningStreaming`) rather than as a separate message kind or
a separate projection collection.

Rationale:

- The collapsible "thinking" block renders directly above its answer, so
  co-locating reasoning with the answer message makes web rendering trivial and
  ordering automatic.
- Reuses the existing assistant **segment** machinery
  (`getOrCreateAssistantMessageId`, `AssistantSegmentState`): reasoning deltas
  resolve to the same per-turn message id as the answer, so reasoning naturally
  attaches to its answer segment. Interleaved reasoning across tool calls lands
  on the active segment — matching the ChatGPT/Claude "Thought for Xs per
  response" UX.
- Avoids touching the message-array invariants relied on by `projector` retention
  logic, `hasAssistantMessageForTurn`, deletion, etc. that a new message kind
  would disturb.

**Known v1 simplification:** all reasoning within one answer segment consolidates
into that message's `reasoningText`. A richer per-step model (Model 2: reasoning
as ordered standalone entries) is deferred; document it as a follow-up if users
want fully interleaved reasoning/tool/reasoning ordering.

**Toggle is client-side display only.** The server always captures reasoning;
`reasoningDisplay` controls visibility and default expand state in the web.
Thinking tokens are generated by the model regardless of the toggle, so
suppressing transport for `off` is not worth the complexity in v1. (Documented
assumption — revisit if bandwidth matters.)

## Layer-by-layer scope

### 1. `packages/contracts/src/orchestration.ts`

- `OrchestrationMessage` (≈ line 227): add
  - `reasoningText: Schema.optional(Schema.String)`
  - `reasoningStreaming: Schema.optional(Schema.Boolean)`
- `ThreadMessageSentPayload` (≈ line 899): mirror the two optional fields so the
  projector can round-trip them, OR (preferred, lower blast radius) add dedicated
  reasoning commands + payload so the answer `message-sent` event is untouched:
  - `ThreadMessageReasoningDeltaCommand` { type, commandId, threadId, messageId,
    delta, turnId?, createdAt } — mirror of `ThreadMessageAssistantDeltaCommand`
    (≈ line 712).
  - `ThreadMessageReasoningCompleteCommand` — mirror of
    `ThreadMessageAssistantCompleteCommand` (≈ line 723).
  - Domain event payload `ThreadMessageReasoningPayload` { threadId, messageId,
    turnId, reasoningText, reasoningStreaming, createdAt, updatedAt }.
- Register the new commands in the command union and the event in the event union
  (follow how the assistant delta/complete are registered — grep both literals).

### 2. `packages/contracts/src/settings.ts`

- Define `ReasoningDisplayMode = Schema.Literals(["off", "collapsed", "expanded"])`.
- `ClientSettingsSchema` (line 42): add
  `reasoningDisplay: ReasoningDisplayMode.pipe(Schema.withDecodingDefault(Effect.succeed("collapsed")))`.
  `DEFAULT_CLIENT_SETTINGS` (line 98) derives automatically.
- `ClientSettingsPatch` (line 541): add
  `reasoningDisplay: Schema.optionalKey(ReasoningDisplayMode)`.
- `UnifiedSettings`/`DEFAULT_UNIFIED_SETTINGS` (453/454) pick it up via the
  intersection — verify no manual default list needs the new key.

### 3. `apps/server/src/orchestration/Layers/ProviderRuntimeIngestion.ts`

- In the main per-event handler (next to `assistantDelta`, ≈ line 1360), add a
  `reasoningDelta` branch:
  `event.type === "content.delta" && event.payload.streamKind === "reasoning_text"`
  (also accept `"reasoning_summary_text"` → same channel for v1).
- Resolve the message id via the **same** `getOrCreateAssistantMessageId` so
  reasoning co-locates with the turn's answer segment; `rememberAssistantMessageId`.
- Respect delivery mode: reuse the buffered/streaming pattern
  (`appendBufferedAssistantText`/spill or direct dispatch) — parameterize the
  existing helpers for a reasoning buffer, OR add a sibling
  `appendBufferedReasoningText`. Dispatch `thread.message.reasoning.delta`.
- Emit `thread.message.reasoning.complete` when reasoning ends:
  - pi: on `thinking_end` (PiDriver should forward a completion signal — see
    note below) or at `message_end`/turn end.
  - Codex: when the reasoning item completes / first answer text arrives.
  - Fallback: flush + complete reasoning at the same points assistant text is
    flushed/completed for the turn.

### 4. `apps/server/src/orchestration/decider.ts`

- Add `case "thread.message.reasoning.delta"` and `".complete"` (mirror lines
  605 / 632): emit a `thread.message-reasoning-delta` domain event with
  `reasoningStreaming: true` / a `...-complete` (or single event with a flag)
  carrying `reasoningStreaming: false`.

### 5. `apps/server/src/orchestration/projector.ts`

- Add a `case "thread.message-reasoning-delta"` reducer (mirror the
  `thread.message-sent` branch at line 389): find the message by id; if absent,
  create a stub assistant message (empty `text`, `streaming: true`,
  `reasoningStreaming: true`) so reasoning that precedes the answer still renders;
  if present, append to `reasoningText` (streaming) and set
  `reasoningStreaming` per the event.
- Ensure the answer `thread.message-sent` branch preserves existing
  `reasoningText`/`reasoningStreaming` when merging (spread already does; verify
  it doesn't blank them).

### 6. `apps/web/src/components/chat/MessagesTimeline.tsx`

- New `ReasoningBlock` component: collapsible disclosure rendered **above** the
  assistant answer body (assistant render path ≈ line 427 / `AssistantMessageBody`
  ≈ line 583). Renders `message.reasoningText` via `ChatMarkdown`.
  - Header: spinner + "Thinking…" while `reasoningStreaming`; else
    "Thought for {duration}" where duration = `updatedAt − createdAt` of the
    message (derive in web, no extra state).
  - Default open state from `reasoningDisplay`: `expanded` → open;
    `collapsed` → closed; `off` → not rendered.
  - While streaming with `collapsed`/`expanded`, auto-expand; collapse to summary
    when reasoning completes (only if mode is `collapsed`).
- Read `reasoningDisplay` from the settings hook used elsewhere in web
  (grep `useSettings`/`DEFAULT_UNIFIED_SETTINGS` usage; mirror how
  `enableAssistantStreaming`/`timestampFormat` are consumed).
- Do not render an empty answer bubble when only reasoning exists yet (guard on
  `message.text` length, reusing the existing `(empty response)` logic at line
  583 so a reasoning-only in-flight message shows just the thinking block).

### 7. `apps/web/src/components/settings/SettingsPanels.tsx`

- Add a tri-state control for `reasoningDisplay` (segmented control / select)
  near `enableAssistantStreaming` (≈ line 665). Wire dirty-state/reset following
  the existing `enableAssistantStreaming` pattern (lines 410, 441, 465, 651-665).

## Driver note (verify, may be no-op)

PiDriver already maps `thinking_delta` → `reasoning_text` (line 428). Confirm it
also surfaces a reasoning **completion** signal (pi `thinking_end`) so ingestion
can fire `reasoning.complete` promptly rather than waiting for turn end. If not,
fall back to completing reasoning at message/turn end — acceptable for v1.

## Decisions (locked)

1. **Default mode** — `collapsed`. (User will flip to `expanded` locally for
   testing.)
2. **`reasoning_summary_text`** — fold into the same reasoning block. Note: only
   the Codex-direct adapter ever emits it; PiDriver (Opus 4.8 / GPT 5.5 via pi)
   emits only `reasoning_text`. Folding is a one-clause condition in ingestion —
   a no-op for the current pi setup, free future-proofing if the Codex provider
   is added later.
3. **Buffered mode** — reuse `enableAssistantStreaming` to decide live-vs-end
   delivery of reasoning (buffered → emit once at completion; streaming → live
   deltas).
